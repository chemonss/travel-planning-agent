# Учебный кейс: Агент для планирования путешествий

Синтетический датасет для учебного E2E-кейса по созданию агента для планирования путешествий. Студенты разрабатывают агента, который подбирает оптимальные билеты, отели и туры с учётом состава группы, бюджета, ограничений и предпочтений путешественников, а также умеет перестраивать маршрут и рекомендации при изменении вводных со стороны пользователя.[file:1]

Решение должно использовать промпты, инструменты доступа к БД с предложениями перелётов, отелей и туров, а также инструменты доступа к профилям путешественников, где предпочтения и ограничения частично заданы в структурированном виде, а частично — в виде текстовых комментариев.[file:1]

## Структура репозитория

```text
data/
  documents/         # Правила сервиса, тарифные ограничения, справки для агента
  travelers/         # БД путешественников и групп путешествий (SQLite)
  reference/         # Эталонные планы, размеченные ответы и сценарные результаты
  qa/                # Q&A датасет для E2E-оценки + описание категорий
  generation/        # Скрипты воспроизводимой генерации синтетики (для преподавателя)
README.md
```

## Документы (RAG)

Рекомендуется включить несколько документов, по которым работает агент:

1. `01_booking_rules.md` — правила бронирования, возвратов и изменений.
2. `02_baggage_and_fares.md` — ограничения тарифов, багаж, пересадки, ночные перелёты.
3. `03_hotel_policy.md` — правила по отелям, категориям, отменам, питанию, детям.
4. `04_package_tours.md` — правила подбора туров, трансферов и включённых услуг.
5. `05_agent_behavior.md` — регламент общения, эскалации, уточнения и пересчёта вариантов.

В документах полезно специально заложить коллизии и исключения: например, тариф дешевле, но без багажа; семейный запрос допускает пересадку не более 3 часов; для группы с ребёнком запрещены ночные прилёты; часть отелей не допускает бесплатную отмену.[file:1]

## База путешественников и предложений (tools)

В `data/travelers/` размещается база `travelers.sqlite`.

Рекомендуемые таблицы:
- `travelers` — профиль путешественника: возраст, гражданство, домашний аэропорт, лояльность, базовые ограничения.
- `traveler_preferences` — предпочтения и комментарии: любит пляжный отдых, избегает ранних вылетов, нужен завтрак, предпочитает 4*+, не любит долгие пересадки.
- `travel_groups` — состав группы, бюджет, целевое направление, даты, ограничения по детям/визам.
- `group_members` — связь группы с путешественниками.
- `flights` — варианты перелётов с ценами, тарифами, багажом, временем вылета и пересадками.
- `hotels` — варианты отелей с ценой за ночь, категорией, питанием, рейтингом, политикой отмены.
- `tours` — пакетные туры, если сценарий включает готовые турпродукты.

По аналогии с шаблонным кейсом полезно иметь сценарные группы путешественников, для которых заранее гарантированы определённые профили и конфликтующие предпочтения, чтобы использовать их в QA и эталонной оценке.[file:1]

## Reference-датасет

В папке `data/reference/` рекомендуется хранить несколько типов эталонных файлов:

- `flight_recommendations.csv` — эталонные рекомендации по перелётам для сценарных групп.
- `hotel_recommendations.csv` — эталонные рекомендации по отелям.
- `trip_plans_gold.jsonl` — эталонные полные планы поездки.
- `change_requests_gold.jsonl` — эталонные реакции системы на изменения параметров поездки.
- `categories.md` — описание типов кейсов и правил оценки.

Минимальная схема для эталонного плана поездки:

| Поле | Описание |
|---|---|
| `group_id` | Идентификатор группы путешествия |
| `scenario_id` | Идентификатор сценария |
| `destination` | Итоговое направление |
| `recommended_flight_id` | Выбранный перелёт |
| `recommended_hotel_id` | Выбранный отель |
| `recommended_tour_id` | Выбранный тур, если применимо |
| `estimated_total_price` | Итоговая оценка стоимости |
| `decision_rationale` | Краткое объяснение выбора |
| `status` | `approved` / `clarification` / `escalation` / `rejected` |

## Q&A датасет (E2E-оценка)

Как и в шаблонном кейсе, основной E2E-набор удобно хранить в `data/qa/qa.jsonl`, где каждый кейс описывает пользовательский запрос, доступный контекст и ожидаемый тип исхода.[file:1]

Рекомендуемые категории:

| Категория | Что проверяет |
|---|---|
| `info` | Ответы по правилам тарифов, отелей и путешествий |
| `planning` | Построение полного плана поездки под бюджет и группу |
| `replanning` | Перестройка варианта после изменения дат, бюджета или состава группы |
| `preference_conflict` | Обработка конфликтующих предпочтений внутри группы |
| `budget_limit` | Корректное поведение при нехватке бюджета |
| `clarification` | Запрос недостающих данных перед рекомендацией |
| `escalation` | Передача на оператора при невозможности автоматически принять решение |
| `edge_case` | Пограничные случаи: визы, дети, ночные рейсы, отсутствие подходящих отелей |

Поле `expected_outcome_type` рекомендуется задавать как `info`, `recommendation`, `clarification`, `escalation` или `rejection`, по аналогии с исходным учебным README.[file:1]

## Воспроизводимость

Вся синтетика должна быть воспроизводима фиксированным seed, а генерация и валидация должны быть вынесены в отдельные скрипты, как в исходном шаблоне с БД клиентов и Q&A-валидацией.[file:1]

Примерный workflow:

```bash
# Перегенерировать БД путешественников и предложений
python3 data/generation/01_generate_travelers_and_inventory.py

# Сгенерировать эталонные reference-файлы
python3 data/generation/02_generate_reference.py

# Сгенерировать и проверить Q&A сценарии
python3 data/generation/03_generate_qa.py
python3 data/generation/04_validate_qa.py
```

Скрипты генерации и сценарные карты предназначены для преподавателя. Студентам передаются только итоговые артефакты в `data/documents/`, `data/travelers/travelers.sqlite`, `data/reference/` и `data/qa/`.[file:1]

## Что использовать на каком этапе

| Этап работы студента | Артефакты |
|---|---|
| Построение RAG | `data/documents/` |
| Построение tools | `data/travelers/travelers.sqlite` + схема |
| Отладка рекомендаций | `data/reference/flight_recommendations.csv`, `hotel_recommendations.csv` |
| Тестирование агента | свои кейсы или подмножество `data/qa/qa.jsonl` |
| Финальная E2E-оценка | весь `data/qa/qa.jsonl` + gold reference-файлы |

Метрики рекомендуется считать отдельно по типам задач: точность выбора перелёта и отеля, корректность укладывания в бюджет, качество учёта предпочтений, корректность запроса уточнений и корректность реакции на изменения требований.[file:1]
